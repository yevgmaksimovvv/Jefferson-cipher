# Архитектура

Jefferson Cipher Service использует многослойную архитектуру для разделения ответственности и обеспечения поддерживаемости.

## Слои

- **Domain**: Чистая бизнес-логика (алгоритм шифрования, модели). Ноль зависимостей от внешних фреймворков или баз данных.
- **Schemas**: Модели Pydantic для валидации запросов и ответов.
- **API Endpoints**: Роуты FastAPI. Обрабатывают только HTTP-задачи.
- **Services**: Код-связка между API/Бизнес-логикой и репозиториями.
- **Repositories**: Слой доступа к базе данных.
- **DB Models**: Модели SQLAlchemy.
- **Alembic**: Миграции базы данных.

## Ограничения (Constraints)

- **Domain** НЕ ДОЛЖЕН импортировать `FastAPI`, `SQLAlchemy`, `Alembic`, `app.api` или `app.core`.
- **API** НЕ ДОЛЖЕН содержать логику шифрования.
- **Repositories** НЕ ДОЛЖНЫ ничего знать об HTTP.
- **Services** НЕ ДОЛЖНЫ зависеть от объектов `Response` FastAPI.

## Жизненный цикл инициализации

Сервис `backend-init` обеспечивает согласованное состояние базы данных перед запуском приложения `backend`:
1. Postgres здоров (healthy).
2. `alembic upgrade head`
3. `python -m app.db.init_db` (наполнение начальными данными/seeding).

Оба сервиса используют один и тот же тег образа `jefferson-cipher-backend` для обеспечения совместимости.
